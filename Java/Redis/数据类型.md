# Redis数据类型
Redis 提供的数据类型并非单纯的逻辑结构，每种逻辑数据类型在 Redis 底层都对应着一种或多种物理数据结构（编码方式），Redis 会根据数据量的大小自动进行编码转换，以在时间效率和空间效率之间取得平衡。

---

## 一、 String（字符串）

String 是 Redis 最基本的数据类型，是二进制安全的，这意味着它可以包含任何数据，如序列化后的 JSON 对象或 JPEG 图像。一个 String 类型的 value 最大可以容纳 512MB。

### 1. 底层编码实现
String 的底层并没有直接使用 C 语言的传统字符串，而是自己构建了一种名为 **SDS（Simple Dynamic String，简单动态字符串）** 的抽象类型。
* **`int` 编码：** 当存储的值是整数，且可以用 long 类型表示时，直接将整数值存在指针位置，节省内存。
* **`embstr` 编码：** 当存储的字符串长度小于等于 44 字节时使用。内存分配只需一次，SDS 结构体和实际字符串在内存中是连续的。
* **`raw` 编码：** 当存储的字符串长度大于 44 字节时使用。需要两次内存分配，SDS 结构体和字符串实体在内存中不连续。



### 2. 核心操作指令
* `SET key value` / `GET key`：基础读写。
* `SETNX key value`：仅当 Key 不存在时设置值（常用于基础分布式锁的实现）。
* `SETEX key seconds value`：设置值的同时指定过期时间。
* `INCR key` / `DECR key`：原子性递增/递减操作。

### 3. 典型应用场景
* **对象缓存：** 将 Java 对象序列化为 JSON 字符串存储。
* **计数器：** 利用 `INCR` 命令的原子性，实现文章阅读量、视频播放量、点赞数的统计。
* **分布式 Session：** 在 Spring Boot 集群中集中存储用户的登录状态。

## 二、 Hash（哈希表）

Hash 是一个键值对集合，类似于 Java 中的 `Map<String, Object>`。特别适合用于存储对象，且允许对对象的单个字段进行独立读写，避免了提取和覆盖整个对象的网络和序列化开销。

### 1. 底层编码实现
* **`ziplist`（压缩列表）：** 当 Hash 中包含的键值对数量较少（默认 < 512 个），且所有键值对的字符串长度都较短（默认 < 64 字节）时使用。它是一块连续的内存，内存利用率极高，但修改效率较低。（注：Redis 7.0 之后引入了 `listpack` 替代 `ziplist`）。
* **`hashtable`（字典）：** 当不满足上述条件时，退化为哈希表。其结构与 Java 的 HashMap 类似，通过数组 + 链表解决哈希冲突，采用渐进式 rehash 机制避免扩容时的主线程阻塞。



### 2. 核心操作指令
* `HSET key field value` / `HGET key field`：设置/获取 Hash 表中的单个字段。
* `HMSET key field1 value1 field2 value2`：批量设置多个字段。
* `HGETALL key`：获取该 Key 下所有的字段和值。
* `HINCRBY key field increment`：对 Hash 表中的某个整数字段进行原子递增。

### 3. 典型应用场景
* **结构化对象存储：** 存储用户详细信息（如 `HSET user:1001 name "Alice" age 25`）。
* **购物车：** Key 为用户 ID，Field 为商品 ID，Value 为商品数量，可通过 `HINCRBY` 直接修改数量。

## 三、 List（列表）

List 是简单的字符串列表，按照插入顺序排序。可以通过命令将元素推入列表的头部（左边）或尾部（右边）。

### 1. 底层编码实现
* **`quicklist`（快速列表）：** 早期的 Redis 使用 `ziplist` 和 `linkedlist`（双向链表）。自 Redis 3.2 起，List 统一采用 `quicklist` 实现。它实际上是一个由 `ziplist` 组成的双向链表。即链表的每个节点不再是一个单纯的元素，而是一个包含多个元素的连续内存块（ziplist），这既保留了链表在两端插入/删除的高效性，又缓解了纯链表指针带来的内存开销和内存碎片问题。



### 2. 核心操作指令
* `LPUSH key value` / `RPUSH key value`：从左侧（头部）或右侧（尾部）推入元素。
* `LPOP key` / `RPOP key`：从头部或尾部弹出元素。
* `LRANGE key start stop`：获取指定范围内的元素列表（常用于分页）。
* `BRPOP key timeout`：阻塞式从尾部弹出元素，若列表为空则阻塞等待直到超时或有新元素。

### 3. 典型应用场景
* **消息队列：** 利用 `LPUSH` + `BRPOP` 实现简单的生产者-消费者模型。
* **信息流展示：** 存储用户最新的关注动态、系统通知等按时间线排序的数据。

## 四、 Set（集合）

Set 是 String 类型的无序集合。集合成员是唯一的，不能出现重复数据。Redis 提供了非常丰富的集合间操作指令。

### 1. 底层编码实现
* **`intset`（整数集合）：** 当集合中的所有元素都是整数，且元素数量较少（默认 < 512 个）时使用。内存空间紧凑，查找采用二分查找。
* **`hashtable`（字典）：** 当元素不是整数或数量超过阈值时，底层使用 Hash 表实现，只不过字典的 value 全部为 null，仅使用 key 来保证唯一性。

### 2. 核心操作指令
* `SADD key member1 member2`：向集合添加一个或多个成员。
* `SMEMBERS key`：获取集合中的所有成员。
* `SISMEMBER key member`：判断元素是否是集合的成员（时间复杂度 O(1)）。
* `SINTER key1 key2` / `SUNION` / `SDIFF`：计算多个集合的交集、并集、差集。
* `SRANDMEMBER key [count]`：随机返回集合中的一个或多个元素。

### 3. 典型应用场景
* **去重与标签系统：** 存储用户的所有标签，或统计某篇文章所有点赞的用户 ID 以实现去重。
* **社交网络关系：** 共同好友（求交集 `SINTER`）、可能认识的人（求差集 `SDIFF`）。
* **随机抽奖：** 利用 `SRANDMEMBER` 随机抽取中奖用户。

---

## 五、 ZSet（有序集合 / Sorted Set）

ZSet 与 Set 类似，也是不允许重复元素的 String 集合。不同之处在于，ZSet 的每个元素都会关联一个 `double` 类型的浮点数分数（Score）。Redis 正是通过这个分数来为集合中的成员进行从小到大的排序。

### 1. 底层编码实现
* **`ziplist`（压缩列表）：** 当元素数量较少（默认 < 128 个）且所有元素长度较短（默认 < 64 字节）时使用。元素和分数紧凑排列在一起。
* **`skiplist`（跳跃表） + `hashtable`（字典）：** 这是 ZSet 最核心的数据结构。
  * `hashtable` 维护了 元素 -> 分数 的映射关系，使得查询某个元素的分数时间复杂度为 O(1)。
  * `skiplist` 维护了 分数 -> 元素 的多层级有序链表。跳表通过在普通链表上增加多级索引，实现了平均时间复杂度为 O(log N) 的快速范围查找和插入。



### 2. 核心操作指令
* `ZADD key score member`：向有序集合添加一个元素及其分数。
* `ZRANGE key start stop [WITHSCORES]`：按分数从低到高返回指定范围的元素。
* `ZREVRANGE key start stop [WITHSCORES]`：按分数从高到低返回指定范围的元素（常用于排行榜查询）。
* `ZSCORE key member`：获取指定元素的分数。
* `ZRANK key member`：返回指定元素的排名（从 0 开始）。

### 3. 典型应用场景
* **排行榜系统：** 游戏积分榜、微博热搜榜、直播间礼物贡献榜（将积分或热度值作为 Score）。
* **延时队列：** 将任务的执行时间戳作为 Score，后台定时轮询提取 Score 小于当前时间的任务进行处理。
* **滑动窗口限流：** 以时间戳为 Score 记录每次请求，根据当前时间往前推一个时间窗口，统计该区间内的记录数量进行频率限制。
