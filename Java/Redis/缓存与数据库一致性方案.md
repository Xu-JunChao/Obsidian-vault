# 缓存与数据库一致性方案 (Cache Aside Pattern)



在微服务架构中，保障 MySQL 与 Redis 之间的数据一致性是核心设计难点。业界标准方案为 **Cache Aside Pattern（旁路缓存模式）**。

## 1. 核心读写逻辑
* **读请求：** 优先查询 Redis 缓存，若命中则直接返回；若未命中，则查询 MySQL 数据库，随后将结果回写至 Redis，最后返回数据。
* **写请求：** **先更新 MySQL 数据库，更新成功后，再主动删除 Redis 中对应的缓存项。**

## 2. 核心原理解析
* **为什么是“删除”而不是“更新”缓存？** 这体现了**懒加载思想**。某些缓存数据的构建逻辑可能较为复杂，关联多张表。如果每次修改数据库都触发缓存的重新计算与更新，在“写多读少”的场景下会造成极大的算力浪费。直接删除缓存，由下一次真正的读请求来触发缓存重建，是最经济的做法。
* **为什么是“先更新 DB，再删除缓存”而不是反过来？**
  如果先删除缓存再更新 DB，在高并发场景下：线程 A 删除了缓存，还没来得及更新 DB；此时线程 B 发起读请求，发现缓存为空，便去读取了 DB 中的旧数据并写入缓存；随后线程 A 才完成 DB 的更新。这会导致缓存中永远是旧数据（即数据不一致）。
