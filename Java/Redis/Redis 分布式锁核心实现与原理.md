# Redis 分布式锁核心实现与原理

在分布式系统中，需利用 Redis 充当分布式锁管理器，防止多个微服务实例对同一共享资源的并发操作冲突。

## 1. 基础实现原理
* **核心指令：** `SET resource_name my_random_value NX PX 30000`
* **`NX` (Not Exists)：** 保证仅当 Key 不存在时才能设置成功，实现排他性（互斥锁）。
* **`PX 30000`：** 设置锁的自动过期时间（如 30 秒），防止持有锁的节点突然宕机而导致锁无法释放（死锁）。
* **唯一标识 (my_random_value)：** Value 必须是当前线程的唯一标识（如 UUID）。在释放锁时，必须验证 Value 匹配才能执行 `DEL`，防止误删其他线程持有的锁。

## 2. 锁超时问题与 Redisson 机制

* **痛点：** 若业务逻辑执行时间较长，超出了锁的设定有效时间，锁会被 Redis 自动释放，导致其他线程趁机获取锁，破坏互斥性。
* **生产级解决方案 (Redisson Watch Dog)：** 在 Spring Boot 项目中通常引入 Redisson 客户端来实现。Redisson 内置了“看门狗 (Watch Dog)”机制。当线程成功获取锁后，看门狗会启动后台定时任务（默认锁租期为 30s，每 10s 执行一次），自动检测并延长该锁的过期时间。只要业务逻辑尚未执行完毕且服务实例存活，锁就不会因超时而失效；若服务异常宕机，看门狗线程随之消亡，锁会在倒计时结束后自然释放。
