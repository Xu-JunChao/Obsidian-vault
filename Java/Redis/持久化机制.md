# Redis 持久化机制：RDB 与 AOF



## 1. RDB (Redis DataBase)
* **机制原理：** **内存快照**。将某一时刻的内存数据全量生成二进制文件 (`.rdb`) 并持久化至磁盘。
* **触发方式：** 手动执行 `save` (阻塞主线程) 或 `bgsave` (Fork 子进程异步执行)；也可通过配置文件（如 `save 60 10000`）自动触发。
* **优势：** 备份文件紧凑体积较小，灾难恢复时加载速度极快。
* **劣势：** 无法做到秒级持久化，两次快照之间的突发宕机会导致这段时间的数据丢失。

## 2. AOF (Append Only File)
* **机制原理：** **命令日志**。将每一条导致状态变化的写命令以追加的方式写入日志文件 (`.aof`) 中。
* **刷盘策略：** `always` (每次写入都刷盘，安全但慢)、`everysec` (每秒异步刷盘，推荐)、`no` (交由操作系统决定)。
* **优势：** 数据安全性较高，通常最多丢失 1 秒内的数据。
* **劣势：** 日志文件体积易膨胀（系统会通过 AOF Rewrite 机制重写文件以瘦身），恢复数据时需重新执行日志中的命令，速度相对较慢。

## 3. 生产环境标准实践
业界通常采用 **RDB 与 AOF 混合持久化**（Redis 4.0 引入）。在 AOF 重写时，将当前数据以 RDB 格式写入 AOF 文件开头，后续新增的写命令再以 AOF 格式追加。利用 RDB 保障恢复速度，利用 AOF 保障数据完整性。
