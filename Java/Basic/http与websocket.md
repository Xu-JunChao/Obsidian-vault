# HTTP 与 WebSocket 详解及核心区别

## 一、 HTTP (HyperText Transfer Protocol) 概念解析

**HTTP（超文本传输协议）** 是万维网数据通信的基础协议。它基于一种经典的“请求-响应” (Request-Response) 模型。



* **单向通信（半双工）：** 通信只能由客户端（通常是浏览器）主动发起。客户端向服务器发送请求，服务器处理后返回响应。在标准的 HTTP 协议中，服务器**绝对不能**主动向客户端推送数据。
* **无状态 (Stateless)：** HTTP 本身是不保存状态的。每一次请求对于服务器来说都是完全独立的新请求。服务器处理完并返回结果后，连接通常就会关闭（虽然 HTTP/1.1 引入了 Keep-Alive 来复用 TCP 连接，但其本质依然是“客户端不问，服务器就不答”）。
* **形象类比：** 就像**发电子邮件**。你发一封邮件给对方，对方收到后给你回一封。如果你不发邮件，对方通常没法直接通过这个渠道实时塞给你信息。

## 二、 WebSocket 概念解析

**WebSocket** 是一种在单个 TCP 连接上进行**全双工 (Full-Duplex)** 通信的协议。它的诞生正是为了弥补 HTTP 在“实时双向通信”方面的致命短板。



* **双向实时通信：** 客户端和服务器只需要进行一次初始的“握手”（这次握手其实是借用了 HTTP 协议来完成的），随后两者之间就会建立起一条持久的专属通道。在这个通道里，**服务器可以随时主动向客户端推送数据**，客户端也可以随时发数据给服务器。
* **持久连接 (Persistent)：** 只要通道建立，除非客户端或服务器其中一方主动断开，否则连接会一直保持打开状态。
* **低数据开销：** 握手成功之后，双方后续传输的数据包头部（Header）非常非常小（通常只有几个字节），这极大地减少了不必要的网络带宽消耗，非常适合高频实时的数据交互。
* **形象类比：** 就像**打电话**。电话一旦拨通（握手成功），双方就可以随时说话，甚至可以同时说话，而且不需要每次开口前都重新拨号。

## 三、 HTTP 与 WebSocket 的核心区别对比

| 对比维度 | HTTP | WebSocket |
| :--- | :--- | :--- |
| **通信模式** | 单向的“请求-响应”模式 (半双工) | 双向的持续通信模式 (全双工) |
| **状态与连接** | 无状态，默认短连接（请求完即断开/复用机制被动） | 有状态，持久长连接（握手后一直保持） |
| **服务器主动性** | **不能**主动推送数据（只能等待客户端请求） | **可以**随时主动向客户端推送数据 |
| **头部开销** | 每次请求都必须携带包含完整信息的庞大 HTTP Header，开销大 | 仅初次握手有 HTTP Header，后续数据帧头部极小，开销极低 |
| **典型适用场景** | 网页浏览、传统 REST API 接口调用、文章加载、表单提交等 | 即时聊天应用 (微信/Slack)、网页多人游戏、股票/加密货币实时 K 线图、在线协同文档等 |
