# Spring 事务机制与传播行为



Spring 通过 AOP 机制实现了对数据库事务的声明式管理（`@Transactional` 注解），极大简化了传统的手动 `commit` 和 `rollback` 代码。

## 1. 事务失效的经典场景
面试中常考 `@Transactional` 为何会失效，本质原因是 **AOP 动态代理机制未生效**：
* **方法非 public：** Spring 默认只会对 public 方法进行代理拦截。
* **同类内部方法调用（自调用）：** 对象内部方法 A 直接调用方法 B 时，是通过 `this.B()` 调用的，绕过了 Spring 生成的代理类，因此方法 B 上的事务注解失效。
* **异常被吞（Catch）：** 业务代码 `try-catch` 了异常且没有向外抛出，导致 Spring 代理类无法捕获异常，从而不会触发回滚。默认情况下，Spring 仅在遇到 `RuntimeException` 或 `Error` 时回滚（可通过 `rollbackFor = Exception.class` 修改）。

## 2. 核心传播行为 (Propagation)
当一个事务方法被另一个事务方法调用时，必须指定事务如何传播。七种传播行为中最核心的三种：
* **`REQUIRED`（默认）：** 如果当前存在事务，则加入该事务；如果当前没有事务，则创建一个新事务。（最常用）。
* **`REQUIRES_NEW`：** 无论当前是否存在事务，都会创建一个**全新的事务**，并将当前事务挂起。新事务与老事务相互独立，外部事务失败回滚不会影响内部已提交的新事务。
* **`NESTED`：** 如果当前存在事务，则在嵌套事务（Savepoint 机制）内执行。内部事务回滚到保存点，不影响外部事务；但外部事务回滚会导致内部事务也回滚。
